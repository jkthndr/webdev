Design File Format Research

For an agent-native design tool where Claude Code creates, iterates, and exports production-ready web UI. Research date: 2026-03-30.

The Core Question

Should the design document be an intermediate format (JSON schema that gets compiled to code), or should designs BE code (the design IS React/TSX + Tailwind, rendered directly)?

The Spectrum of Approaches

→

The further right you go, the smaller the fidelity gap between design and production code. The tradeoff is structured manipulability.

Key Insight from Research

The industry is converging toward "designs as code". Paper.design chose HTML/CSS as both medium and output. Google Stitch exports HTML/CSS/React directly. v0.dev generates React+Tailwind+shadcn as its native output. The intermediate-format approach (Figma-style) is increasingly seen as a liability because of the translation gap.

However, Pencil's .pen format demonstrates that a well-designed JSON schema with proper MCP tooling can give agents excellent ergonomics for structured manipulation while still being renderable. The question is whether that structured manipulation advantage outweighs the cost of maintaining a JSON-to-code compiler.

Reference: Excalidraw's Format

Excalidraw uses a clean JSON structure optimized for simplicity. Each element has a type, position, dimensions, and style properties.

{
  "type": "excalidraw",
  "version": 2,
  "elements": [
    {
      "id": "abc123",
      "type": "rectangle",       // rectangle | ellipse | diamond | text | arrow | line | freedraw | image | frame
      "x": 100, "y": 50,
      "width": 200, "height": 120,
      "strokeColor": "#1e1e1e",
      "backgroundColor": "transparent",
      "fillStyle": "solid",      // solid | hachure | cross-hatch
      "strokeWidth": 2,
      "roughness": 1,            // hand-drawn effect intensity
      "opacity": 100,
      "angle": 0,
      "groupIds": [],
      "frameId": null,
      "roundness": { "type": 3 },
      "seed": 1234,              // deterministic randomization
      "version": 1,
      "versionNonce": 5678       // conflict detection
    }
  ],
  "appState": {
    "viewBackgroundColor": "#ffffff",
    "gridSize": 20,
    "zoom": { "value": 1 },
    "theme": "light"
  },
  "files": {}                    // FileId -> BinaryFileData for images
}

Key features: soft-delete via isDeleted, arrow bindings via startBinding/endBinding, text containers via containerId. Round-trip embedding in PNG (iTXt chunks) and SVG (metadata comments).

Reference: Pencil's .pen Format

Pencil's format is the most sophisticated JSON schema studied. It's a full design system format with variables, themes, components/instances, flexbox layout, and rich graphics.

{
  "version": "2.9",
  "themes": { "device": ["phone", "tablet", "desktop"] },
  "variables": {
    "primary-color": { "type": "color", "value": "#FF6B6B" },
    "spacing-sm":    { "type": "number", "value": 8 }
  },
  "children": [
    {
      "id": "frame1",
      "type": "frame",           // frame | group | rectangle | ellipse | line | polygon | path | text | icon_font | ref
      "name": "Login Screen",
      "layout": "vertical",      // none | vertical | horizontal
      "width": 390, "height": "fit_content(844)",
      "gap": 16,
      "padding": [24, 24, 24, 24],
      "justifyContent": "center",
      "alignItems": "center",
      "fill": "#FFFFFF",
      "cornerRadius": 12,
      "children": [
        {
          "id": "title1",
          "type": "text",
          "content": "Welcome Back",
          "fontSize": 28,
          "fontWeight": "700",
          "fill": "$primary-color",   // variable reference with $ prefix
          "textGrowth": "fixed-width",
          "width": "fill_container"
        },
        {
          "id": "btn1",
          "type": "ref",             // component instance
          "ref": "ButtonComp",       // references reusable component by ID
          "descendants": {
            "label": { "content": "Sign In" }
          }
        }
      ]
    }
  ]
}

Key features: ref nodes for component instances with deep descendant overrides, $variable binding syntax, theme-aware variables, fill_container / fit_content dynamic sizing, slot system for component customization. Encrypted on disk, only accessible via MCP tools.

Reference: Figma's Node Model

Figma uses a mixin-based property system with 38 scene node types. Key properties organized by category:

// Figma Node Type Hierarchy
BaseNode = DocumentNode | PageNode | SceneNode

SceneNode types (38): FrameNode, RectangleNode, TextNode, ComponentNode,
  InstanceNode, EllipseNode, LineNode, VectorNode, GroupNode, SectionNode,
  BooleanOperationNode, StarNode, PolygonNode, SliceNode, ...

// Properties via Mixins (composition over inheritance)
GeometryMixin:  x, y, width, height, rotation, absoluteBoundingBox, absoluteTransform
LayoutMixin:    layoutMode, itemSpacing, padding*, primaryAxisAlign*, counterAxisAlign*,
                layoutSizingHorizontal, layoutSizingVertical, layoutGrow
StyleMixin:     fills[], strokes[], effects[], opacity, blendMode
CornerMixin:    cornerRadius, topLeftRadius, ...cornerSmoothing
TextMixin:      characters, fontName, fontSize, fontWeight, textAlign
ChildrenMixin:  children[], findAll(), findOne(), findChildren()
ComponentMixin: componentPropertyDefinitions, addComponentProperty()

// Notable: getCSSAsync() returns computed CSS for any node

Figma's REST API returns the full document as JSON. Its Plugin API uses TypeScript typings. The mixin pattern means not all nodes share all properties -- FrameNode has ChildrenMixin, RectangleNode does not.

Pros
  • Highly structured -- agent can query/modify any node by path
  • Batch operations are natural (update all buttons, restyle all cards)
  • Clean diffs in git (JSON is line-diffable)
  • Design system tokens are first-class (variables, themes)
  • Schema validation catches errors before rendering
  • Framework-agnostic: can compile to React, Vue, HTML, etc.
  • Pencil proves this works well with MCP tooling
Cons
  • Must build and maintain a JSON-to-code compiler
  • Fidelity gap: compiled output never matches hand-written code exactly
  • Must build a custom renderer for visual preview
  • Two sources of truth: the schema AND the generated code
  • Agent must learn a custom DSL (not transferable knowledge)
  • Schema evolves independently of web standards (maintenance burden)
  • Real-world CSS edge cases are hard to model in a schema

Approach 2: HTML/CSS (Paper.design's Model)

How It Works

Agent writes HTML/CSS
→
Browser renders directly
→
Export = the HTML itself

Paper.design's Philosophy

"If the canvas is built on the same standards as the product -- HTML, CSS, DOM -- then you're not drawing a metaphor of a UI. You're working in the medium."

-- Paper.design founding philosophy

Paper.design chose HTML/CSS as both the design medium and the output format. Key architectural decisions:

  • The canvas speaks HTML/CSS natively -- layout uses real Flexbox (and planned CSS Grid)
  • Visual effects are GPU-rendered but map to CSS properties
  • 24 MCP tools give agents full bidirectional read/write access
  • Agents can convert designs to React/Tailwind and commit to GitHub
  • Can import and use actual React components on the canvas

What the Agent Works With

<!-- Agent writes/modifies this directly -->
<div style="display: flex; flex-direction: column; gap: 16px;
            padding: 24px; background: #FFFAF5; border-radius: 12px;
            max-width: 390px; margin: 0 auto;">
  <h1 style="font-family: Inter, sans-serif; font-size: 28px;
             font-weight: 700; color: #2D2A26; margin: 0;">
    Welcome Back
  </h1>
  <div style="display: flex; flex-direction: column; gap: 8px;">
    <label style="font-size: 14px; color: #6B6560;">Email</label>
    <input type="email" placeholder="you@example.com"
           style="padding: 12px; border: 1px solid #E8E0D8;
                  border-radius: 8px; font-size: 16px;" />
  </div>
  <button style="padding: 14px; background: #FF6B6B; color: white;
                 border: none; border-radius: 8px; font-size: 16px;
                 font-weight: 600; cursor: pointer; width: 100%;">
    Sign In
  </button>
</div>

Agent Ergonomics Challenge

The main problem: HTML is verbose and unstructured for programmatic manipulation. To change all button colors, the agent must:

// Agent needs to: find all buttons, parse their inline styles, update the color
// There's no structured way to do "update all elements where role=button"
// unless you add a query layer on top

// Contrast with JSON:
// nodes.filter(n => n.type === "button").forEach(n => n.style.background = "#E8773A")
Pros
  • Zero fidelity gap -- what you see IS the output
  • No renderer to build (browser is the renderer)
  • No exporter to build (HTML is the export)
  • Agents already know HTML/CSS well
  • Git-diffable (though diffs can be noisy)
  • Paper.design validates this works commercially
  • Can convert to JSX/React as a post-processing step
Cons
  • Verbose: inline styles are repetitive and token-heavy
  • No structured node tree for programmatic manipulation
  • Batch operations require DOM parsing (regex on HTML is fragile)
  • No design system tokens as first-class concept
  • No component abstraction (copy-paste, not reuse)
  • Hard to enforce consistency across screens
  • HTML diffs are noisy (attribute order, whitespace)

Approach 3: React/TSX ("Designs ARE Code")

How It Works

Agent writes TSX + Tailwind
→
Dev server renders
→
Screenshot for preview
→
Export = the TSX itself

v0.dev Validates This Approach

Vercel's v0 generates React + Tailwind + shadcn/ui directly from prompts. The output IS production code. There is no intermediate format -- the model writes TSX, and that TSX is what ships. This has become the dominant paradigm for AI-generated UI in 2025-2026.

What the Agent Works With

// LoginScreen.tsx -- this IS the design AND the production code
import { Button } from "@/components/ui/button"
import { Input } from "@/components/ui/input"
import { Label } from "@/components/ui/label"

export function LoginScreen() {
  return (
    <div className="flex flex-col gap-4 p-6 bg-[#FFFAF5] rounded-xl max-w-[390px] mx-auto">
      <h1 className="text-[28px] font-bold text-[#2D2A26]">
        Welcome Back
      </h1>
      <div className="flex flex-col gap-2">
        <Label htmlFor="email" className="text-sm text-[#6B6560]">
          Email
        </Label>
        <Input
          id="email"
          type="email"
          placeholder="you@example.com"
          className="border-[#E8E0D8]"
        />
      </div>
      <Button className="w-full bg-[#FF6B6B] hover:bg-[#FF5252] text-white">
        Sign In
      </Button>
    </div>
  )
}

Why This Works for AI Agents

Claude Code already writes React/TSX + Tailwind as its primary output mode. The agent doesn't need to learn a custom DSL, custom schema, or custom tooling. It uses the same skills it uses for every other coding task. The design system is shadcn/ui -- which the agent already knows deeply.

Structured Operations via AST

The "no structured manipulation" objection is partially mitigated by the fact that TSX is parseable. The agent can:

// Batch restyle: agent can find-and-replace Tailwind classes
// "Change all bg-[#FF6B6B] to bg-[#E8773A]" -> simple text replacement

// Structured via AST parsing (if needed):
// - Parse TSX with @babel/parser
// - Walk the AST to find all <Button> elements
// - Modify their className prop
// - Serialize back to TSX

// But in practice, the agent just... edits the file. It understands the structure.

Rendering Pipeline

// Option A: Running dev server + Playwright screenshot
// 1. Agent writes/modifies .tsx files
// 2. Next.js dev server hot-reloads
// 3. Playwright navigates to the route and screenshots
// 4. Agent sees the screenshot, iterates

// Option B: Isolated component rendering
// 1. Agent writes a component to a preview route
// 2. e.g. /preview/login-screen renders just that component
// 3. Screenshot via headless browser

// Option C: Static rendering (no server needed)
// 1. Use react-dom/server to renderToStaticMarkup
// 2. Inject into an HTML template with Tailwind CDN
// 3. Screenshot the static HTML
Pros
  • ZERO fidelity gap -- the design literally IS the production code
  • No translation, no compiler, no exporter to build or maintain
  • Agent uses existing skills (React, Tailwind, shadcn) -- no custom DSL
  • shadcn/ui gives you a real component system with variants and slots
  • Git-diffable, code-reviewable, CI-testable
  • v0.dev proves this model works at scale commercially
  • Design tokens via Tailwind config (colors, spacing, typography)
  • Component reuse is native (just import and use)
  • Can add interactivity directly (hover states, animations, transitions)
  • Entire ecosystem of tooling already exists
Cons
  • Needs a running dev server or static renderer for visual preview
  • Batch operations less natural than JSON (but AST parsing helps)
  • Multi-file management (components, layouts, pages) adds complexity
  • Heavier setup than a single JSON file
  • Tailwind class strings can get long and verbose
  • Agent needs to manage imports and file structure

How It Works

→
Compiler generates TSX
→
Dev server renders
→
Export = compiled TSX

Builder.io as Reference

Builder.io uses this exact approach. Their internal format is a JSON block tree:

{
  "@type": "@builder.io/sdk:Element",
  "id": "builder-abc123",
  "tagName": "div",
  "layerName": "Login Card",
  "responsiveStyles": {
    "large": { "display": "flex", "flexDirection": "column", "gap": "16px", "padding": "24px" },
    "medium": { "padding": "16px" },
    "small": { "padding": "12px" }
  },
  "children": [
    {
      "@type": "@builder.io/sdk:Element",
      "component": { "name": "Text", "options": { "text": "Welcome Back" } },
      "responsiveStyles": { "large": { "fontSize": "28px", "fontWeight": "700" } }
    },
    {
      "@type": "@builder.io/sdk:Element",
      "component": { "name": "Button", "options": { "text": "Sign In" } },
      "responsiveStyles": { "large": { "width": "100%" } }
    }
  ],
  "bindings": {},
  "actions": {},
  "repeat": null
}

This JSON compiles to React, Vue, Angular, Svelte, or plain HTML depending on the target framework. The visual editor works with the JSON; the output is framework code.

Pencil as Reference

Pencil's .pen format is essentially this approach too: a rich JSON schema with a visual renderer. The difference is Pencil renders to its own canvas rather than compiling to framework code.

The JSON-to-React Compiler Challenge

// You must handle ALL of these:

// 1. Layout mapping
{ layout: "vertical", gap: 16, padding: 24 }
// -> className="flex flex-col gap-4 p-6"

// 2. Design token resolution
{ fill: "$primary-color" }
// -> className="bg-primary"  (if mapped to Tailwind config)
// -> style={{ background: tokens.primary }}  (if runtime)

// 3. Component mapping
{ type: "component", componentRef: "Button", props: { variant: "primary" } }
// -> <Button variant="primary">...</Button>

// 4. Responsive styles
{ responsiveStyles: { large: { fontSize: "28px" }, small: { fontSize: "20px" } } }
// -> className="text-xl md:text-[28px]"

// 5. Edge cases: animations, gradients, mesh gradients, SVG paths, image fills...
// Each adds compiler complexity
Pros
  • Best of both: structured JSON for manipulation, real code for output
  • Excellent batch operations (JSON is trivially queryable)
  • Framework-agnostic (compile to React, Vue, HTML, etc.)
  • Clean separation of concerns
  • Pencil and Builder.io prove it works commercially
Cons
  • Must build and maintain the JSON-to-code compiler -- this is the #1 cost
  • Compiler is a permanent fidelity gap -- compiled code is never as clean as hand-written
  • Two representations to keep in sync (schema version + compiler version)
  • Agent learns a custom DSL that doesn't transfer to other work
  • Every new CSS feature or component pattern requires compiler updates
  • Builder.io has a large team maintaining their compiler; we don't

How Existing Tools Handle This

Figma

Proprietary Binary REST API returns JSON Plugin API with TypeScript

Figma stores designs in a proprietary binary format internally. The REST API exposes the document as a JSON tree of typed nodes (38 SceneNode types). Properties are organized via mixins (GeometryMixin, LayoutMixin, StyleMixin, etc.).

Agent access: Figma MCP gives agents read access to the node tree, design tokens, and layout constraints. However, Figma MCP has only 3 tools and provides read-only access. Developers report 85-90% styling inaccuracy when converting Figma designs to code.

Key lesson: Figma's format is optimized for visual editing, not code generation. The translation gap is its biggest weakness for AI workflows.

Paper.design

HTML/CSS Native 24 MCP Tools Bidirectional Agent Access

The canvas IS HTML/CSS. Designs use real Flexbox, real CSS. The 24-tool MCP server gives agents full read/write access. get_jsx returns JSX output for any node. Export quality is inherently better than Figma MCP because there's no proprietary format to translate from.

Key lesson: Using the production medium (HTML/CSS) as the design medium eliminates the translation gap entirely. Paper proves this works for both human designers and AI agents.

Pencil (.pen)

Encrypted JSON Schema Rich MCP Tooling Component System

Pencil uses a rich JSON schema (v2.9) with: typed nodes (frame, text, rectangle, ellipse, path, icon_font, ref), flexbox layout, design variables with $ binding, theme-aware values, component instances via ref with deep descendant overrides, slot system, and image fills.

MCP tools include batch_get (search/read nodes), batch_design (insert/copy/update/replace/move/delete in batch), get_screenshot, snapshot_layout. The batch operations syntax is a mini-DSL: card=I(parent, {...}), U(card+"/title", {...}).

Key lesson: A well-designed schema with purpose-built MCP tools gives excellent agent ergonomics. The batch operation syntax is compact and powerful. BUT: the format is encrypted/proprietary and requires Pencil's renderer for preview -- you can't just open it in a browser.

Penpot

SVG Native Open Source Clojure/ClojureScript

Penpot stores designs as SVG -- every shape IS an SVG node with Penpot metadata augmentation. A "Shape" is the core entity, corresponding to one SVG element. Properties include: geometry (selrect, x, y, w, h), transforms (2D matrix), constraints, interactions, fills/strokes/shadows/blur, and text-specific attributes.

Penpot files export as compressed archives containing SVG + JSON. SVG import is bidirectional: Penpot exports reconstruct perfectly, third-party SVGs get best-effort conversion.

Key lesson: Using an open standard (SVG) as the native format gives interoperability and eliminates vendor lock-in. However, SVG is not ideal for web UI design (no flexbox, no responsive layout).

Google Stitch

Cloud AI Canvas DESIGN.md Exports HTML/CSS/React

Stitch takes a unique approach with DESIGN.md -- an agent-friendly markdown file that captures design rules (colors, typography, spacing, component patterns). When you submit a prompt, Stitch passes both the prompt and DESIGN.md to Gemini as context. The model uses the design rules to constrain its output.

DESIGN.md format is simple markdown with specific values:

## Colors
- Primary: #1A73E8
- Background: #FFFFFF
## Typography
- Font Family: Inter, sans-serif
- Heading 1: 32px, 700 weight
## Spacing
- Base unit: 8px
## Components
- Button border radius: 8px
- Card shadow: 0 1px 3px rgba(0,0,0,0.12)

Stitch exports to Figma, Google AI Studio, Jules, or downloadable HTML/CSS/React code.

Key lesson: DESIGN.md as a design system contract is a clever idea. It's human-readable, agent-parseable, git-versionable, and works across tools. It separates design tokens from design implementation.

v0.dev

React + Tailwind + shadcn/ui Code-First No Intermediate Format

v0 generates React/TSX + Tailwind + shadcn/ui directly from natural language prompts. There is no intermediate format. The AI model writes production code. Output can be copied, installed via CLI into any Next.js project, or iterated on with follow-up prompts.

Key lesson: The most commercially successful AI design tool uses NO intermediate format. The design IS the code. This is the strongest evidence that "designs as code" works at scale.

Excalidraw

Plaintext JSON Simple Schema MCP Available

Simple JSON format with elements array. Each element: type, position, dimensions, style properties. Features: soft-delete, arrow bindings, text containers, deterministic seeding for hand-drawn rendering. Round-trip embedding in PNG/SVG. MCP server available for agent manipulation.

Key lesson: Excalidraw's format is optimized for whiteboarding/diagramming, not UI design. Too low-level for our use case (no components, no layout system, no design tokens). But its simplicity and JSON-first approach are instructive.

Recommendation: React/TSX with a Thin Schema Layer

Primary format: React/TSX + Tailwind + shadcn/ui

The design document should BE React code. The agent writes TSX files using Tailwind CSS and shadcn/ui components. The rendering is done by a dev server (or static renderer). The export is the TSX itself.

Why React/TSX Wins

  1. Zero fidelity gap -- the single most important criterion. The mockup literally IS the production code. No translation, no drift, no "it looked different in the design tool."
  2. Agent already knows it -- Claude Code writes React/Tailwind/shadcn every day. No custom DSL to learn, no new schema to memorize. This matters enormously for agent quality.
  3. Proven at scale -- v0.dev, Bolt.new, and Lovable collectively prove this model works commercially. They all generate React+Tailwind directly.
  4. Nothing to build -- no schema, no validator, no renderer, no compiler, no exporter. All the tooling already exists. Focus engineering effort on the agent experience, not format plumbing.
  5. Full design system -- Tailwind config provides design tokens (colors, spacing, typography, breakpoints). shadcn/ui provides a complete component library with variants. This is a better design system than any custom JSON schema.
  6. Ecosystem leverage -- ESLint, Prettier, TypeScript type-checking, Playwright for screenshots, Storybook for component isolation, Git for versioning. All free.

Add a Thin Metadata Layer (Not a Full Schema)

To address the "batch operations" weakness, add a lightweight metadata file alongside the TSX -- not a full intermediate format, but a project manifest:

// design.config.json -- project manifest, NOT the design itself
{
  "project": "MyApp",
  "screens": [
    { "id": "login", "file": "screens/LoginScreen.tsx", "viewport": "mobile" },
    { "id": "dashboard", "file": "screens/Dashboard.tsx", "viewport": "desktop" },
    { "id": "settings", "file": "screens/Settings.tsx", "viewport": "mobile" }
  ],
  "designSystem": {
    "tailwindConfig": "tailwind.config.ts",
    "components": "components/ui/",
    "tokens": {
      "colors": { "primary": "coral-500", "surface": "warmgray-50" },
      "fonts": { "sans": "Inter" }
    }
  },
  "exportSettings": {
    "framework": "nextjs",
    "styling": "tailwind",
    "componentLibrary": "shadcn"
  }
}

This manifest gives the agent structured access to project metadata without duplicating the actual design content in a second format.

Borrow DESIGN.md from Stitch

Also include a DESIGN.md file (inspired by Google Stitch) as a human+agent-readable design system contract:

- Primary: coral-500 (#FF6B6B) -- buttons, links, accents
- Surface: warmgray-50 (#FFFAF5) -- page backgrounds
- Text Primary: warmgray-900 (#2D2A26)
- Text Muted: warmgray-500 (#6B6560)

## Typography
- Font: Inter, system-ui fallback
- H1: 28px / 700 weight
- Body: 16px / 400 weight
- Caption: 12px / 400 weight

## Spacing
- Base: 4px grid
- Component gap: 16px (gap-4)
- Section padding: 24px (p-6)
- Page margin: 32px (p-8)

## Components
- Buttons: rounded-lg, font-semibold
- Cards: rounded-xl, shadow-sm, border border-warmgray-200
- Inputs: rounded-lg, border border-warmgray-300

## Principles
- Warm palette only -- no blue/purple
- Light/pastel theme, soft shadows
- Rounded corners everywhere (min 6px)

Handling Batch Operations

The main weakness of the code-first approach. Mitigations:

Operation How It Works in React/TSX
Restyle all buttons All buttons use <Button> from shadcn -- update the component definition once. Or find-replace the Tailwind class.
Change primary color Update tailwind.config.ts -- one line change propagates everywhere via utility classes.
Change font Update tailwind.config.ts font family -- propagates via font-sans class.
Add a field to every form Agent uses grep to find all form components, edits each. More work than JSON batch, but agent handles it.
Query: "which screens use the Card component?" grep -r "import.*Card" screens/ -- standard code search.
Move a section across screens Cut-paste the JSX block. Components are self-contained.

What Not to Build

  • Do not build a JSON-to-React compiler. This is the biggest trap. Builder.io has a large engineering team maintaining theirs. The compiler is never done -- every new CSS feature, every new component pattern, every edge case requires compiler work.
  • Do not build a custom renderer. The browser is the renderer. A Next.js dev server or static HTML is the preview.
  • Do not build a custom schema language. TypeScript IS the schema. TSX IS the format. Tailwind config IS the design tokens.
  • Do not encrypt/obfuscate the format. Pencil's encrypted .pen files require custom tooling for every operation. TSX files are readable by any editor, diffable by any tool, parseable by any AST library.

Implementation Path

PhaseWhatEffort
1. Core Agent writes TSX + Tailwind to a project directory. Screenshot via Playwright. Iterate based on visual feedback. Low
2. Design System Scaffold a project with tailwind.config.ts, shadcn/ui components, and DESIGN.md. Agent reads DESIGN.md for every design task. Low
3. Preview Pipeline Set up preview routes (e.g., /preview/[screen]) that render individual screen components. Playwright screenshots each. Medium
4. Project Manifest Add design.config.json for screen inventory, viewport info, and export settings. Agent reads this to understand project structure. Low
5. Component Library Agent builds reusable components in components/, uses them across screens. Same as normal React development. Low

Key Takeaway

The best design format for an AI agent is the format the agent already produces.

Claude Code writes React/TSX + Tailwind + shadcn/ui. Making the design format match the agent's native output eliminates an entire class of problems: no schema to design, no compiler to build, no renderer to maintain, no fidelity gap to debug. The agent does what it already does best, and the output is directly usable.

The intermediate-format approach (JSON schema) made sense when the primary user was a human designer who needed visual manipulation tools. For an agent-native tool where the primary creator IS an AI that thinks in code, the code-first approach is strictly superior.
