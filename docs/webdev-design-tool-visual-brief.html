Strategy

What this tool is, and what it is not

Open Design has good workflow ideas. webdev has a better production-fidelity foundation. The win is combining those without cloning the broader product.

This is webdev

Self-hosted web UI design-to-code

An agent takes a brief, creates real React/TSX screens, shows a live preview from the running app, captures Playwright approval screenshots, checkpoints with git, and hands off the code.

  • Next.js + Tailwind + shadcn/ui output
  • Studio for screenshots, comments, diffs, and approval
  • MCP/API surface for agents
This is not webdev

Not a general creative studio

Do not chase decks, office docs, image generation, video, audio, or a Figma-style vector editor. Those broaden the surface before the core loop is even lovable.

  • No slide deck generator
  • No generic HTML artifact playground
  • No bundled brand-replica design systems
Borrow the workflow

Guided brief, skills, run progress

Open Design proves that users need taste scaffolding: short discovery, design-system context, style recipes, visible run status, and critique before approval.

  • Structured brief before generation
  • User-owned DESIGN.md
  • Design lint and review gates
Keep the edge

Do not lose the TSX source of truth

The strongest webdev differentiator is that the design is already the code. The product should make that loop easier to drive, not introduce a separate design format.

  • Code is the deliverable
  • Screenshots prove render fidelity
  • Git checkpoints make iteration safe

Rendering decision

Live preview is real app preview

Studio can embed an iframe, but that iframe should load /proxy/<project>/screens/<screen>, which proxies the running Next app. Playwright screenshots are still generated after build/start and become the approval, diff, QA, and handoff artifact.

Commercial safety

Clean-room rules

The goal is to learn from the market while keeping webdev original enough for a future paid product.

Never copy

Protected product material

No Open Design source code, prompts, skill bodies, template files, design-system markdown, generated examples, assets, or UX copy.

Allowed

General product patterns

It is fine to independently implement common ideas: guided brief capture, run history, screenshot review, design lint, and read-only handoff tools.

Original content

Templates and style recipes

Every bundled template, style recipe, prompt, and sample screen should be authored for webdev and carry provenance metadata.

User content

Imports and design systems

User-provided screenshots, imported ZIPs, and company DESIGN.md files can guide a project, but should not become redistributed boilerplate.
