Paper paper.design — 24 MCP Tools (11 Read / 8 Write / 5 Utility)

Paper is an HTML/CSS DOM-native design tool with GPU shader effects. Its MCP server provides bidirectional access — agents can both read and modify the live canvas. Key differentiator: write_html for direct HTML injection, get_jsx for Tailwind/inline-style code export, and working indicators for human awareness.

1. Common Patterns (Table Stakes)

These capabilities appear across all three platforms. Any design MCP server must have them.

Every design MCP server provides:

  • Document/project creation — Pencil: open_document("new"), Paper: create_artboard, Stitch: create_project
  • Visual preview / screenshot — All three: get_screenshot / get_screen_image. Critical for the review-iterate loop. The agent must see what it built.
  • Node/element inspection — Pencil: batch_get, Paper: get_node_info + get_children, Stitch: get_screen
  • Code/markup export — Paper: get_jsx, Stitch: get_screen_html. (Pencil lacks this, agents must infer code from node data.)
  • Design modification — Pencil: batch_design, Paper: write_html + update_styles, Stitch: edit_screen
  • Canvas space management — Pencil: find_empty_space_on_canvas, Paper: find_placement. (Stitch handles this implicitly.)

2. Unique Strengths

Pencil's Unique Strengths

  • Batch scripting DSL — The I()/C()/U()/R()/D()/M()/G() syntax is the most expressive write interface. Agents can build complex layouts in one call with variable bindings across operations.
  • Atomic rollback — If any operation in a batch fails, all previous ops roll back. Safe for complex multi-step edits.
  • Layout problem detection — snapshot_layout(problemsOnly: true) finds clipping, overlap, and misalignment without vision. No other tool has this.
  • Bulk property operations — search_all_unique_properties + replace_all_matching_properties enable design-system-wide refactoring (swap all brand colors, change all font sizes).
  • Design guidelines system — get_guidelines gives agents task-specific instructions and style archetypes. Built-in prompt engineering for design tasks.
  • AI image generation in batch — G(nodeId, "ai", prompt) generates and places images inline with design operations.

Paper's Unique Strengths

  • HTML/CSS as design language — write_html means agents write the same code they already know. No proprietary format to learn. The DOM IS the design.
  • Built-in JSX/Tailwind export — get_jsx(format: "tailwind") gives production-ready React code directly. Zero design-to-code translation gap.
  • Working indicators — start/finish_working_on_nodes tells the human when the agent is actively modifying. Unique collaborative awareness.
  • Font validation — get_font_family_info checks font availability before use. Prevents rendering issues.
  • Computed styles access — get_computed_styles reads resolved CSS properties, not just declarations. More accurate than reading source styles.
  • Largest tool surface — 24 tools means more granular operations. Each tool does one thing well.

Stitch's Unique Strengths

  • AI-first generation — generate_screen_from_text creates entire screens from prompts. Highest abstraction level. Agent describes, AI designs.
  • Variant exploration — generate_screen_variants with creativeRange and aspects parameters enables systematic design exploration. No manual A/B needed.
  • Design DNA extraction — extract_design_context pulls fonts, colors, layouts from existing screens to seed new generation. Cross-screen consistency via AI.
  • Multi-page site assembly — build_site maps screens to routes, outputting a complete multi-page site. End-to-end generation.
  • Rich skills layer — 7 stitch-skills (prompt enhancement, React conversion, video generation) show how to compose tools into complete workflows.
  • Model selection — Agents can choose between GEMINI_3_PRO (quality) and GEMINI_3_FLASH (speed) per generation.

3. Gaps — Underserved Agent Workflows

Gaps across ALL three platforms:

  • No version history / undo for agents — None offer undo(), checkpoint(), or restore(version). Agents can't safely experiment and roll back. Pencil has atomic rollback within a batch, but no cross-batch undo.
  • No responsive preview — None offer "render at mobile/tablet/desktop" from a single design. Stitch has deviceType on generation, but no responsive reflow of an existing screen.
  • No accessibility audit — None have built-in contrast ratio, ARIA, or focus-order checking. Agents must infer accessibility from visual/structural data.
  • No design-to-component-library mapping — None map design elements to a specific component library (shadcn, Material, etc.) structurally. Paper gets closest with get_jsx, but it outputs raw HTML/Tailwind, not component imports.
  • No interactive prototype / state management — None handle hover states, click interactions, page transitions, or conditional rendering in the design layer.
  • No collaborative diff / merge — No tool supports "compare version A with version B" or "merge these two design variants."
  • No design linting / guidelines enforcement — Pencil has get_guidelines but no automated enforcement. No tool can lint_design(rules) to check spacing consistency, color system compliance, etc.
  • No real-time human feedback channel — Paper's working indicators are closest. But there's no request_human_review() that pauses the agent and waits for approval.
  • No cross-platform export — None export to Flutter, SwiftUI, or Kotlin Compose. All are web-only (React/HTML/CSS).

Design Decisions & Rationale

Why batch_modify uses JSON, not a scripting DSL like Pencil ▶

Pencil's I()/C()/U()/R()/D()/M() DSL is powerful but requires agents to learn a custom syntax. JSON operations are universally parseable, easier to validate, and work natively with LLM tool-calling conventions. We keep the same expressiveness (insert, copy, update, replace, delete, move) but in structured form. Atomic rollback is preserved.

Why we include BOTH batch_modify AND write_html ▶

Paper proves that write_html is extremely efficient for agents — they already know HTML/CSS. But batch_modify gives fine-grained control for updates, moves, and deletes that HTML injection can't express. Different tools for different phases: write_html for initial layout, batch_modify for precise iteration.

Why checkpoint/restore instead of undo ▶

Agents don't think in sequential undo steps. They think in versions: "before I tried this approach" vs "after." Named checkpoints let agents safely experiment and return to known-good states. This fills the biggest gap in all three existing platforms.

Why lint_design is a first-class tool ▶

Pencil's snapshot_layout(problemsOnly) proves agents need automated QA. We extend this with contrast checking (WCAG), spacing consistency, color system compliance, and typography hierarchy validation. This replaces manual "look at the screenshot and check for issues" with structured issue reports the agent can systematically fix.

Why we skip AI generation tools (like Stitch) ▶

Our agent IS the AI. Stitch uses a separate AI model (Gemini) to generate screens. We don't need that layer — Claude is the agent doing the designing, using our MCP tools as its hands. Adding AI generation inside the tool creates a confusing two-model architecture. Instead, we give the agent precise tools and let it use its own intelligence to compose designs.

Why export_code supports react-shadcn format ▶

Paper exports raw JSX/Tailwind. Stitch exports raw HTML/CSS. Neither maps to a real component library. Our react-shadcn format outputs code that uses actual shadcn/ui imports (<Button>, <Card>, <Input>) instead of generic divs. This is the gap that matters most for production code quality.

Why responsive viewport on get_screenshot ▶

No existing tool lets you screenshot the same design at different viewport sizes. Our viewport parameter renders with responsive CSS reflow, so agents can verify mobile/tablet/desktop without maintaining separate designs. This is critical since our designs use Tailwind responsive classes.
