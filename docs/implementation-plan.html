Agent-Native Design Tool Implementation Plan

Open-source, self-hosted alternative to Pencil — where designs ARE production code

Option C: Designs Are Code 2-3 Week MVP 7 MCP Tools (MVP) React + Playwright + shadcn/ui

Executive Summary

This tool makes Claude Code a first-class design agent. The agent writes real React/TSX components using shadcn/ui and Tailwind CSS, a Next.js dev server renders them, Playwright captures screenshots for human review, and the TSX source IS the production deliverable — zero translation gap.

After evaluating 5 competing approaches (extending Penpot, building from scratch, wrapping Stitch, hybrid approaches), Option C: "Designs ARE Code" wins decisively: 10/10 fidelity, 2-3 week MVP, minimal maintenance, zero vendor lock-in.

Recommended: Build a 7-tool MVP MCP server (expanding to 11+ in later phases) that manages design projects as Next.js apps. Validate with 3 spikes before building. Deploy on the HP laptop (jake-dev-win-hp / 100.102.138.90) via Docker with GitHub Actions CI/CD. Strictly code-first — no independent design schema.

Revised After Third-Party Review — March 2026

System Architecture

Architectural Invariants

Source of truth is TSX/React + Tailwind + shadcn/ui. No independent visual schema. No design document compiler. Node operations are AST operations over code.

This is a strict, non-negotiable commitment locked after third-party review. The system will never build a parallel node model, batch_modify DSL over a shadow schema, or write_html injection layer. Every design operation is a code operation. Every code operation produces a real, renderable React component. Violations of this invariant are architectural bugs.

Why This Architecture Wins

The critical insight is eliminating the "design-to-code" translation step entirely. In every other tool (Pencil, Paper, Stitch, Penpot), the design lives in a proprietary format that must be compiled or converted to production code. That conversion is where fidelity is lost.

By making the design format identical to the production format (React/TSX + Tailwind + shadcn/ui), the "export" operation is a no-op. The screenshot you approve IS rendered from the code you ship. There is no shadow schema, no node tree, no intermediate representation — just code.

Operational Specifications

Font Pinning

Bundle Geist Sans and Geist Mono directly in the Docker image. No external font loading at runtime. This ensures deterministic text rendering and consistent screenshot output across environments. Fonts are loaded via next/font/local in the project template.

Checkpoint TTL

Auto-cleanup git checkpoints older than 7 days (configurable via CHECKPOINT_TTL_DAYS environment variable). A background cron job runs daily to prune stale commits. Prevents unbounded disk usage across long-running projects.

Single-Session Constraint

V1 supports one agent session per project at a time. No concurrent agent access to the same project directory. This sidesteps git merge conflicts and file-locking complexity. Multi-session support is a V2+ concern after studying OpenPencil's patterns.

Deterministic Rendering

Fixed viewport dimensions per device class. Preloaded fonts (no FOUT/FOIT). networkidle wait strategy before screenshot capture. No external asset loading at screenshot time. Validated by Spike 3 before build begins.

Resource Contention: Android Build Runner

The HP laptop runs a GitHub Actions runner (hp-android) for Android builds. Gradle builds are memory-intensive (~4-6GB). Mitigations:

  • mem_limit: 1536m on the preview container caps Chromium memory
  • Android builds are infrequent (manual trigger or push to main with [build] tag)
  • Playwright uses a persistent browser context — no cold starts during builds
  • If contention occurs: pause the design tool containers during Android builds, or schedule builds off-hours
  • 32GB RAM total: ~6GB Android build + ~1.5GB design tool + ~1GB TaskQ + OS = ~10GB peak. Plenty of headroom.

Risk Assessment
1

Dual-Architecture Drift

Risk of accidentally building a shadow schema or node model alongside the code-first pipeline. This was the reviewer's primary concern — the original 16-tool set implied parallel internal representations (get_node_tree, batch_modify).

Mitigation: Locked "code-first, no schema" as architectural invariant. Removed all node-centric tools from MVP. Code review explicitly watches for this pattern. Any tool that introduces a non-TSX representation is an architectural bug.

High Impact
2

Deterministic Rendering

Fonts, timing, and asset loading can cause screenshot flakiness. If the same TSX produces different screenshots across runs, the agent's visual feedback loop breaks.

Mitigation: Spike 3 validates this before build begins. Docker image pins all fonts (Geist Sans/Mono). networkidle wait strategy. No external asset loading at screenshot time. Fixed viewport dimensions.

High Impact
3

Agent Code Quality

The agent writes TSX — if it writes broken JSX, invalid imports, or runtime errors, the render fails and the screenshot pipeline stalls.

Mitigation: lint_design tool (V1.1) validates before render. Error boundaries catch runtime failures. Dev server error overlay captured in screenshot so agent sees the error. Checkpoint/restore enables recovery.

High Impact
4

Playwright Resource Usage

Chromium is memory-hungry (~700MB). On the 32GB HP laptop this is fine, but could be an issue on smaller machines.

Mitigation: Persistent browser context — launch once, reuse for all screenshots. Single Chromium instance, not one per screenshot.

Medium
5

HMR Reliability

Next.js hot module replacement must be fast and reliable for the screenshot pipeline. If HMR breaks or stalls, screenshots show stale content.

Mitigation: Turbopack (default in Next.js 16) is significantly faster than Webpack HMR. Add health checks and page-reload fallback if HMR times out.

Medium
6

shadcn/ui Coverage

The MVP component set may not cover all design needs. Some screens may require components not yet in shadcn/ui.

Mitigation: Agent can write custom components alongside shadcn. V1.1 adds component library browsing to see what's available. Custom registry in future.

Low
7

Network Latency

MCP server on HP laptop, Claude Code on Dell laptop. Tailscale adds network hop for every tool call.

Mitigation: Tailscale adds only 1-5ms. Negligible vs. screenshot render time (200-500ms). Can also run via stdio locally.

Low

Date: March 2026

Reviewer: Independent architecture reviewer

Verdict: Direction is strong and well-researched. Biggest risk is accidental dual-architecture. Cut scope to ship faster and de-risk.

Overall Assessment: The product thesis (agent-native, code-first design tool) was validated. The reviewer's concerns focused on execution risk, not strategic direction. All 7 concerns have been addressed in this revised plan.

Concerns and Resolutions

1. Strategy Conflict: "No Intermediate Format" vs Node-Centric MCP

The docs recommend React/TSX as source of truth (good), but proposed MCP tools (get_node_tree, batch_modify, write_html, export_code) imply a parallel internal design model. Risk: you unintentionally build both a code-first system and a schema/translation system.
Resolved: Locked "code-first, no schema" as an architectural invariant. Removed all node-centric tools from MVP. The edit_screen_code tool operates on TSX via AST, not a shadow model. write_html deferred to V2+. export_code deferred to V2+ (in code-first, export is a no-op).

2. react-shadcn Export Is High-Risk

Converting arbitrary HTML/TSX into idiomatic shadcn/ui components is a hard semantic mapping problem. Risk: export quality becomes inconsistent and brittle, especially with nested layouts and variant inference.
Resolved: Deferred export_code to V2+. In the code-first architecture, the agent writes shadcn/ui components directly. There is no conversion step. The TSX IS the deliverable. This concern is eliminated by the architecture itself.

3. Scope Creep in v1 MCP Surface

The proposed 16-tool set is good directionally, but too large for a first reliable release. Risk: many partially-working tools instead of a few excellent ones.
Resolved: Cut MVP from 16 tools to 7. Phased remaining tools into V1.1 (lint_design, bulk_style_replace, design tokens) and V2+ (export_code, write_html). Each phase gates on the previous being solid.

4. Evidence Quality Is Mixed

A lot of sources are community/blog posts; some are likely lower-confidence than official docs/specs. Risk: architecture choices made on noisy data.
Acknowledged: Validation spikes (Phase 0.5) test core assumptions with real code before committing to the build. Fidelity, editability, and determinism are all validated empirically, not based on blog posts.

5. Security Model Is Under-Specified

If the agent can write/render arbitrary code for previews, you need strict execution isolation and resource controls. Risk: unsafe code execution, runaway resource usage, and unclear trust boundaries.
Partially addressed: Docker containerization provides base isolation. Single-session constraint limits blast radius. Resource limits in Docker Compose. Full sandbox model (process-level isolation, CPU/memory caps per render) is documented as a V2 concern.

6. Mobile Goal Is Underdelivered

Docs discuss responsive web screenshots, but "web/mobile UI designs" can imply native stacks. Risk: expectation mismatch unless scope is explicitly "mobile web first".
Resolved: Scope is explicitly responsive web only for V1. The render_screenshot tool supports viewport parameters for mobile/tablet/desktop widths. Native mobile (React Native, Flutter) is a V2+ consideration only.

7. Missing Operational Concerns

The docs under-spec: data/version migrations, checkpoint retention/cleanup, concurrent agent sessions, deterministic rendering (fonts, asset loading, timing). These become painful once used across many projects.
Resolved: Added Operational Specifications section covering font pinning (Geist Sans/Mono bundled in Docker), checkpoint TTL (7-day auto-cleanup), single-session constraint (one agent per project in V1), and deterministic rendering (fixed viewport, preloaded fonts, networkidle wait). Each spec is validated by the corresponding spike.
